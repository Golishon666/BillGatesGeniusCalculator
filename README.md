# Bill Gates Genius Calculator

## Пояснительная записка

Это моя реализация тестового задания с калькулятором в Unity. Я постарался сделать не просто рабочий экран, а небольшое приложение с понятным разделением ответственности: отдельно логика калькулятора, отдельно отображение, отдельно всплывающее окно с ошибкой и отдельно код, который связывает эти части между собой.

Проект открывался и проверялся в Unity `6000.3.14f1`. Основная сцена находится в `Assets/Scenes/SampleScene.unity`.

## Что получилось

В приложении есть экран калькулятора с полем ввода, кнопкой `RESULT` и историей вычислений. Сейчас по условию включено сложение целых неотрицательных чисел: например, `54+21`, `45+00`, `1+2+3`.

Если выражение корректное, я добавляю результат в историю в формате `5+5=10`, очищаю поле ввода и сохраняю состояние. Новые записи появляются сверху.

Если выражение некорректное, в историю добавляется запись вида `выражение=ERROR`, поле временно очищается, показывается модальное окно с сообщением `Please check the expression you just entered`, а после нажатия `GOT IT` введенное выражение возвращается обратно в поле. Так пользователю не приходится вводить все заново.

Также добавил debug-действие: клавиша `Space` очищает историю, если поле ввода сейчас не в фокусе.

## Архитектура

Я опирался на Clean Architecture и MVP, потому что для такого задания это хороший способ не смешивать Unity UI и бизнес-логику.

Проект разбит на несколько assembly:

- `Calculator` - основной модуль калькулятора.
- `MessageBox` - отдельный переиспользуемый модуль модального окна.
- `CalculatorMessageBoxBridge` - небольшой связующий модуль между калькулятором и message box.
- `Calculator.Tests` - EditMode-тесты.

Внутри калькулятора логика тоже разделена:

- `Domain` содержит проверку и вычисление выражений.
- `Application` содержит presenter, состояние, интерфейсы репозитория и сервиса сообщений.
- `Infrastructure` отвечает за сохранение состояния в JSON.
- `Views` содержит Unity-компоненты, которые работают с prefab-объектами.

Важный момент: модуль `Calculator` не знает напрямую про `MessageBox`, а `MessageBox` не знает про `Calculator`. Они общаются через событие и отдельный bridge. Я сделал так, чтобы калькулятор можно было запустить и без конкретной реализации окна ошибки, а message box можно было переиспользовать в другом месте.

## Валидация выражений

Проверка выражения находится в `AdditionExpressionEvaluator`. Я не стал фильтровать ввод во время набора: пользователь может ввести что угодно, а проверка происходит после нажатия `RESULT`. Такой подход ближе к описанному сценарию задания.

Сейчас валидными считаются только выражения из целых чисел и включенных операций. По умолчанию включено только сложение. Остальные операции (`-`, `*`, `/`) уже есть в конфиге `CalculatorOperationsConfig`, но выключены. Это оставляет место для расширения без переделки интерфейсов.

Ошибкой считаются, например:

- пустая строка;
- пробелы;
- дробные числа;
- выражения с `+` в начале или конце;
- повторные операции вроде `1++2`;
- неподдержанные или выключенные операции;
- переполнение числа.

## Сохранение состояния

Я сохраняю текущий ввод и историю через `UnityJsonStateRepository`. Сохранение идет в JSON, а загрузка происходит при старте приложения. Если файла нет или он поврежден, приложение спокойно стартует с пустым состоянием.

## Dependency Injection

Для DI я использовал VContainer.

`CalculatorAppView` наследуется от `LifetimeScope` и регистрирует зависимости калькулятора: evaluator, repository, presenter, view и сервис сообщений. `CalculatorStartup` инициализирует View и запускает восстановление состояния.

`MessageBoxLifetimeScope` регистрирует зависимости окна ошибки и открывает публичный метод `ShowAsync`.

`CalculatorMessageBoxLifetimeScope` хранит ссылки на оба дочерних scope и подписывает калькулятор на показ message box. За счет этого связующий код остается маленьким и не протекает в основной модуль калькулятора.

## Тесты

Я добавил EditMode-тесты для самых важных мест:

- вычисление выражений;
- успешный и ошибочный сценарии presenter;
- сохранение и восстановление состояния;
- поведение сервиса сообщений;
- базовые архитектурные ограничения.

Архитектурные тесты проверяют, что в assembly калькулятора только `View`-классы наследуются от `MonoBehaviour`, а сами View имеют явный метод `Initialize`.

## Почему я сделал именно так

Главная идея была в том, чтобы решение оставалось простым для просмотра, но при этом не превращалось в один большой `MonoBehaviour`. Для тестового задания калькулятор небольшой, но я все равно разнес логику по слоям, потому что так легче проверять поведение тестами и проще расширять проект дальше.

Например, если позже понадобится включить умножение или заменить модальное окно, не придется переписывать presenter или UI с нуля. Достаточно поменять конфиг операций или заменить bridge/реализацию message box.

При этом я не стал усложнять проект сверх необходимости: UI оставлен prefab-based, без генерации всех элементов из кода, а доменная логика написана обычными C#-классами, которые легко читать и тестировать.

## Как запустить

1. Открыть проект в Unity `6000.3.14f1` или совместимой версии Unity 6.
2. Открыть сцену `Assets/Scenes/SampleScene.unity`.
3. Запустить Play Mode.
4. Ввести выражение в поле и нажать `RESULT`.

Для проверки тестов можно открыть Unity Test Runner и запустить EditMode-тесты из `Calculator.Tests`.
